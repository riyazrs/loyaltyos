# LoyaltyOS — Project Documentation
## Human Design + AI Execution: A Collaborative Build Log
### NCI MSc AI for Business | CEAI Final Project

---

## Overview

This document records the full development of LoyaltyOS — a white-label AI loyalty platform — with explicit separation between what was **designed and decided by the human** and what was **generated and executed by AI**. It also documents the iterative validation loop: where the human reviewed AI outputs, identified gaps, and updated the architecture in response.

The purpose of this separation is to demonstrate that agentic AI systems are not autonomous — they are directed, evaluated, and corrected by a human who retains full architectural ownership throughout.

---

## PART 1: HUMAN-DESIGNED ARCHITECTURE

### 1.1 The Core Idea

The human identified the problem: small and mid-size service businesses (pubs, clinics, gyms, salons) cannot afford to build custom loyalty programmes. Existing solutions are generic, expensive, and not AI-powered. The human defined the solution concept:

> A white-label platform where any business submits their details, and AI generates a fully personalised loyalty programme — mechanics, copy, config, and live app — specific to that business.

This concept was not generated by AI. It was the human's original business idea and the foundation every subsequent decision was built on.

---

### 1.2 System Architecture — Human Design Decisions

The human made the following architectural decisions before any AI code was written:

**Decision 1: Two-system architecture**
The human decided to split the system into two distinct components:
- A **Python backend pipeline** (research and strategy generation)
- A **Next.js frontend app** (customer-facing loyalty experience)

These could have been combined into a single system. The human chose to separate them because the pipeline is a one-time generation tool (run when a new business onboards) while the app is a persistent, real-time product.

**Decision 2: Sequential multi-agent pipeline**
The human chose a sequential 5-agent structure — not a single large prompt, not parallel agents. The reasoning was explicit: a real consultancy solving this problem would use specialists working in sequence. Each specialist hands off to the next. The human defined:
- How many agents: 5
- What order they run: Researcher → Designer → Maker → Communicator → Manager
- What each agent's role is
- What each agent's output format should be

This pipeline structure was designed before any code was written, documented in the system design document (`loyaltyos_system_design.md`).

**Decision 3: White-label via config**
The human decided that the platform should be driven by a `BusinessConfig` object — a single data structure that controls everything: colours, currency name, activities, rewards, tiers. Switching from one business to another happens by changing the config, not the code. This is a deliberate architectural constraint that keeps the system extensible.

**Decision 4: Two demo verticals**
The human selected the two demo businesses:
- **PixelPub** (gaming pub) — chosen to demonstrate off-peak revenue mechanics and gamification
- **DentalPlus** (dental clinic) — chosen to demonstrate habit-formation mechanics and appointment management

These were not random. They were chosen to represent two fundamentally different engagement models: discretionary entertainment vs. preventive healthcare.

**Decision 5: LangGraph for orchestration**
The human chose LangGraph specifically for its state persistence, visual graph representation, and production-grade reliability — not just `anthropic.messages.create` in a loop. This was a deliberate technology choice, not an AI suggestion.

**Decision 6: Vercel + GitHub Pages deployment model**
The human decided that new businesses would onboard via a GitHub Pages static form, the form would trigger a GitHub Actions workflow, and the pipeline would run on GitHub's infrastructure — committing the generated config back to the repo and triggering Vercel to redeploy. This three-layer architecture (GitHub Pages → GitHub Actions → Vercel) was the human's design, chosen because it is fully serverless, requires no dedicated backend, and uses infrastructure the human already controlled.

---

### 1.3 Agent Role Definitions — Human Authored

The human wrote all five agent system prompts. These are the most critical human inputs in the entire system — they define each agent's personality, scope, output format, and constraints. A poorly written system prompt produces a generic, useless agent. The human invested significant effort in making each prompt distinct:

| Agent | Human-designed constraint |
|---|---|
| Researcher | "Be specific to the business provided. Do not give generic loyalty programme advice." |
| Designer | "You are obsessed with the moment a customer feels genuinely rewarded." |
| Maker | "This document will be used as the direct build specification by the development team." |
| Communicator | "Each must feel human, not robotic. No corporate speak." |
| Manager | "Score each agent's output (1-10) and give one sentence of honest feedback." |

The Manager's scoring directive was a deliberate human choice to force the pipeline to self-audit. This was not standard practice — it was the human's insight that a manager agent would be most valuable if it held the other agents accountable.

---

### 1.4 Form Design — Human Authored

The human specified exactly what questions the business onboarding form should ask, and why each question matters for the pipeline:

| Question | Why the human included it |
|---|---|
| Business type (9 options) | Determines which engagement mechanics are appropriate |
| Primary customer age range | Shapes persona definitions in the Researcher agent |
| Visit frequency | Critical input for streak and challenge mechanics |
| What customers love | Gives the Designer authentic emotional territory to work with |
| Why customers return | Distinguishes habitual vs. event-driven engagement |
| Primary retention challenge | Directly shapes which AI interventions are prioritised |
| Ideal loyal customer | Gives the Manager a concrete benchmark to evaluate against |

Each of these was a deliberate human decision about what information the AI pipeline needs to produce a specific, non-generic output.

## PART 2: AI CONTRIBUTIONS

### 2.1 Pipeline Execution

Given the human-designed system prompts, agent structure, and state schema, the AI (Claude claude-sonnet-4-6) executed the following for the DentalPlus demo run:

**Researcher agent output:**
- Identified 5 specific behavioural churn signals for dental patients (e.g. no-show without rebooking within 72 hours; appointment gap exceeding 9 months)
- Defined 3 customer personas: "Conscientious Clara", an anxious avoider, and a family-focused parent
- Identified the "invisible progress problem" — good oral hygiene produces no visible reward — as the core engagement challenge
- Mapped competitor landscape: Bupa Dental Care Club, Philips Sonicare app, Starbucks Rewards as analogous
- Flagged GDPR Article 9 risk for health-adjacent data

**Designer agent output:**
- Named the loyalty programme "BrightPath"
- Created "SmilePoints" as the currency
- Designed a 4-tier structure: Starter → Regular → Dedicated → Champion
- Invented "grace day mechanics" — a no-penalty streak recovery system for anxious patients
- Proposed a "Family Oral Health Score" as a household engagement feature
- Designed a surprise-and-delight "Anniversary Smile" milestone card

**Maker agent output:**
- Produced TypeScript interfaces for all data models
- Defined component props for all 6 UI components
- Designed the `/api/personalise` and `/api/offers` Claude API route architecture
- Generated a deployment checklist with environment variables

**Communicator agent output:**
- Wrote the full onboarding sequence (welcome, first reward, 7-day streak messages)
- Produced three promotional offer templates (low-turnover day, tier upgrade, win-back)
- Wrote the B2B sales pitch and GDPR-compliant privacy notice in plain English

**Manager agent output:**
- Scored each agent: Researcher 9/10, Designer 9/10, Maker 8/10, Communicator 9/10
- Identified 4 strategic gaps across the pipeline (see Section 3.2)
- Produced a 3-phase operational roadmap
- Classified the AI personalisation system as "limited risk" under the EU AI Act

### 2.2 Code Generation

The AI generated all implementation code based on the human's architectural decisions:

- `pipeline/agents/*.py` — 5 agent node functions
- `pipeline/pipeline.py` — LangGraph graph assembly
- `pipeline/run.py` — CLI entrypoint
- `pipeline/generate_config.py` — BusinessConfig JSON generator
- `app/src/app/**` — all Next.js pages (landing, dashboard, activities, rewards, admin)
- `app/src/components/**` — all 6 UI components
- `app/src/app/api/**` — 3 API routes (personalise, offers, onboard)
- `.github/workflows/run-pipeline.yml` — GitHub Actions workflow
- `docs/index.html` — GitHub Pages onboarding form

The human reviewed all generated code before it was committed. No code was committed without human review.

## PART 3: ITERATIVE VALIDATION — HUMAN REVIEW CYCLES

This section documents where the human evaluated AI outputs, identified problems, and updated the architecture. This is the most important part of the document — it demonstrates that the human was not a passive observer but an active evaluator and decision-maker throughout.

---

### 3.1 Validation Cycle 1: Model Version

**What happened:** After the pipeline code was written, the human noticed the model ID used was `claude-sonnet-4-20250514` — an outdated string.

**Human decision:** Upgrade all agents and API routes to `claude-sonnet-4-6` (the latest model at time of development).

**Action taken:** Updated all 5 agent files and both Next.js API routes.

**Why this matters:** The human was tracking model versioning independently and caught an error that would have caused silent quality degradation. The AI had no mechanism to self-correct this.

---

### 3.2 Validation Cycle 2: Pipeline Output Review

**What happened:** After running the DentalPlus pipeline, the human reviewed all 5 output files. The Manager agent's quality review surfaced four gaps in the pipeline outputs:

1. The Designer referenced "anxiety-pattern patients" without specifying how a patient gets flagged as anxious
2. The `FamilyDashboard` component was truncated before the children's data handling was defined — a high-risk GDPR gap
3. No API route definitions were produced by the Maker — engineering would need to reverse-engineer these from component props
4. Children's data flow was unspecified despite GDPR-K obligations being flagged by the Researcher

**Human decision:** Rather than re-running the pipeline, the human made a deliberate architectural decision: these gaps would be documented in the submission reflection as evidence of where multi-agent systems require human oversight. This is itself a design decision — choosing transparency over perfection.

**Why this matters:** A human-absent system would have passed these truncated outputs downstream without flagging them. The Manager agent surfaced them, but the human decided what to do about them. The AI identified the problem; the human made the call.

---

### 3.3 Validation Cycle 3: Windows Encoding Issue

**What happened:** When the pipeline was first executed, it crashed with a `UnicodeEncodeError` — the Windows terminal (cp1252 encoding) could not render the emoji characters used in agent progress messages.

**Human decision:** Remove all emoji from print statements and replace with ASCII labels (`[RESEARCHER]`, `[DONE]`, `[ERROR]`, etc.).

**Action taken:** Updated all 5 agent files and `run.py`.

**Why this matters:** The AI had generated code that worked on macOS/Linux but not on the human's Windows 11 environment. The human identified the root cause (encoding, not logic), and the fix was targeted and minimal — not a brute-force rewrite.

---

### 3.4 Validation Cycle 4: Nested Git Repository

**What happened:** When staging the Next.js app files for the initial git commit, git warned that `create-next-app` had initialised its own `.git` repository inside the `app/` directory, creating a nested git submodule conflict.

**Human decision:** Remove the nested `.git` directory from `app/`, unstage, re-stage all files correctly.

**Why this matters:** This was an infrastructure decision — the human understood git's submodule semantics and chose a clean fix over workarounds. The AI had scaffolded the Next.js app correctly per its purpose; the conflict arose at the monorepo level, which was the human's architectural choice.

---

### 3.5 Validation Cycle 5: Business Onboarding Architecture

**What happened:** After the initial build was complete, the human identified a fundamental gap: there was no mechanism for a new business to submit their details and trigger the pipeline. The two demo businesses were hardcoded.

**Human decision:** Design and implement a full onboarding flow — GitHub Pages form → Vercel API → GitHub Actions → pipeline → `dynamicConfigs.json` → Vercel redeploy.

**Architectural choices made by the human:**
- Use GitHub Pages (not a separate frontend) because the repo already had a `docs/` folder
- Use GitHub Actions (not a dedicated server) to run the Python pipeline — free, auditable, version-controlled
- Use a `dynamicConfigs.json` file (not a database) to store generated business configs — keeps the system serverless
- Merge static and dynamic configs in `businessConfig.ts` so new businesses appear at `/{slug}` without any code changes

This entire architecture was the human's response to a gap the human identified. The AI implemented it, but the design was the human's.

---

### 3.6 Validation Cycle 6: Build Memory Error

**What happened:** Running `npm run build` failed with a Node.js out-of-memory error (`AlignedAlloc Allocation failed`) when Turbopack attempted to process the Tailwind CSS worker.

**Human decision:** Increase the Node.js heap size with `NODE_OPTIONS="--max-old-space-size=4096"` rather than downgrading Next.js or switching from Turbopack.

**Why this matters:** The human diagnosed the error correctly (memory allocation, not code logic), chose the minimal intervention (environment variable, not architectural change), and verified the fix produced a clean build before committing.

## PART 6: KEY INSIGHT — WHY HUMAN OVERSIGHT CANNOT BE REMOVED

The Manager agent's quality review (Validation Cycle 2) scored the Communicator agent 9/10 but noted:

> "The persona labelling in Section 1.3 is internally inconsistent — Angela is described as 'the committed patient' when the research clearly positions her as the anxious avoider."

This is a subtle content error — not a code error, not a crash, not a missing field. It is the kind of error that would reach a customer and damage trust. The AI pipeline surfaced it via the Manager agent, but a human must decide whether to re-run, correct manually, or document it as a known limitation. The AI cannot make that decision. The human made it.

This is the core argument for human-in-the-loop oversight in agentic systems: not because AI makes catastrophic errors, but because AI makes subtle, plausible-sounding errors that require human judgment to evaluate and act on.
